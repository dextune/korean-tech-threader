# 공통 스레드 작성 규칙

이 파일은 Korean Tech Threader의 모든 페르소나가 공통으로 따라야 하는 스레드 작성 규칙이다.

페르소나는 말투와 화법을 바꿀 수 있지만,
이 파일의 구조, 검증, 출력 원칙은 바꾸지 않는다.

## 기본 목표

스레드는 정보를 길게 설명하는 글이 아니다.

모바일 환경에서 빠르게 읽히고,
첫 문장에서 멈추게 만들고,
끝까지 따라오게 만드는 짧은 게시물이다.

모든 스레드는 다음 목표를 가진다.

- 첫 문장에서 관심을 끌기
- 짧은 문장으로 리듬 만들기
- 하나의 중심 주장만 다루기
- 독자가 공감할 만한 문제 제기
- 중간에 반전 또는 관점 전환 넣기
- 핵심 문제나 긴장 요소를 분명하게 만들기
- 마지막에 기억나는 결론 남기기

## 작성 절차

스레드를 만들 때는 다음 순서로 생각한다.

1. 사용자가 준 자료의 핵심 주제를 파악한다.
2. 독자가 흔히 믿는 착각이나 놓치는 지점을 찾는다.
3. 그 착각을 깨는 반전 문장을 만든다.
4. 핵심 문제, 근거, 리스크, 기회 중 하나로 현실감을 만든다.
5. 마지막에 하나의 결론으로 닫는다.

## 기본 구조

스레드는 다음 흐름을 따른다.

### 1. 훅

첫 문장은 강하게 시작한다.

좋은 훅은 다음 중 하나를 한다.

- 흔한 착각을 짚는다.
- 지금 많이 놓치는 지점을 말한다.
- 좋아 보이는 선택의 숨은 비용을 보여준다.
- 독자가 이미 겪었을 문제를 먼저 꺼낸다.

구조 예시:

```text
[주제 도입]
[겉으로 좋아 보이는 기준]
[그 기준만 보면 생기는 위험]
```

### 2. 현상 제시

지금 벌어지는 흐름을 짧게 설명한다.

긴 배경 설명은 피한다.
독자가 이미 아는 이야기는 압축한다.

### 3. 반전

중간에는 반드시 관점 전환이 있어야 한다.

구조 예시:

```text
[전환 문장]
```

```text
[처음에는 작아 보이는 문제]
[반복되면 커지는 결과]
```

### 4. 긴장 요소

스레드에는 독자가 계속 읽을 만한 긴장 요소가 들어가야 한다.

긴장 요소는 주제와 페르소나에 맞게 선택한다.

사용할 수 있는 방향:

- 숨은 비용
- 놓치기 쉬운 문제
- 기대와 현실의 차이
- 반복될수록 커지는 영향
- 도입 전에 확인해야 할 조건
- 실패했을 때 드러나는 약점

### 5. 결론

마지막은 짧고 기억나야 한다.

결론은 새로운 주제를 열지 않는다.
앞에서 쌓은 내용을 한 문장으로 닫는다.

구조 예시:

```text
[최종 결론]
[겉으로 보이는 기준이 아니라]
[실제로 중요한 기준]
```

## 문장과 줄바꿈

문장은 짧게 쓴다.

권장 기준:

- 한 문장 12~35자
- 한 줄 12~24자
- 한 문단 2~5줄
- 한 스레드 8~25문장

줄바꿈 기준:

- 강조하고 싶은 단어 앞에서 끊는다.
- 반전 문장은 단독 줄로 둔다.
- 숫자나 핵심 수치는 별도 줄에 둔다.
- 결론 문장은 짧게 단독 처리한다.

나쁜 구조:

```text
[긴 배경 설명과 여러 주제를 한 문장에 모두 넣은 문장]
```

좋은 구조:

```text
[주제]
[겉으로 보이는 시점]
[실제로 중요한 시점]

[처음에는 좋아 보이는 이유]
[반복되면 생기는 문제]
```

## 정보와 주장

스레드는 하나의 메시지만 다룬다.

나쁜 방향:

```text
[주제 A도 중요하고]
[주제 B도 중요하고]
[주제 C도 중요하고]
[주제 D도 중요하다는 식으로 여러 메시지를 동시에 넣기]
```

좋은 방향:

```text
[이번 스레드에서 다룰 하나의 중심 주장]

[다른 요소는 짧게 인정]
[하지만 이번 글의 중심 문제로 다시 회수]
[결론]
```

## 사실 검증

출력의 어조와 관계없이 사실 검증을 우선한다.

다음은 임의로 만들지 않는다.

- 존재하지 않는 모델명
- 기업명
- 보고서명
- 벤치마크 수치
- 날짜
- 통계

최신 정보가 필요한 경우 최신 자료를 확인한다.

출처가 부족하면 단정하지 않는다.

완화 표현의 구체적인 문구는 페르소나 모듈의 말투에 맞춰 선택한다.

## 원문 사용 규칙

참고 자료가 있어도 문장을 그대로 가져오지 않는다.

금지:

- 원문 문장 그대로 사용
- 일부 단어만 바꾼 문장
- 출처가 불분명한 주장을 사실처럼 쓰기
- 같은 수치와 사건을 근거 없이 반복하기

권장:

- 같은 주제를 새로운 순서로 구성하기
- 예시는 다른 상황으로 다시 쓰기
- 기술명과 수치는 실제 자료가 있을 때만 사용하기
- 문체는 참고하되 문장은 새로 만들기

## 입력이 부족할 때

주제만 있고 자료가 부족하면 구체적인 기업명, 모델명, 수치를 만들지 않는다.

대신 페르소나 모듈에 정의된 완화 표현을 사용한다.

자료가 제공된 경우에는 제공된 자료를 우선 사용한다.

## 최종 점검

출력 전에 다음을 확인한다.

- 첫 문장이 스크롤을 멈추게 하는가?
- 문장이 너무 길지 않은가?
- 중간에 반전 문장이 있는가?
- 수치나 사례를 과장하지 않았는가?
- 마지막 문장이 기억나는가?
- 하나의 주장으로 연결되어 있는가?
- 페르소나 말투가 적용되어 있는가?
