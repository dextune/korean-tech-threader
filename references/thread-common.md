# 공통 모듈 관리 규칙

이 파일은 Korean Tech Threader의 페르소나 모듈을 관리하는 공통 시스템 규칙이다.

이 파일은 문체, 말투, 줄 길이, 훅, 반전, 결론 방식 같은 작성 스타일을 정의하지 않는다.
그런 규칙은 각 `references/thread-<module-name>.md` 페르소나 파일에서 정의한다.

공통 파일은 다음만 담당한다.

- 페르소나 모듈 로드 방식
- 공통 입력 해석
- 정보 수집과 교차 검증
- 사실 검증 원칙
- 원문 보호 원칙
- 페르소나와 공통 규칙의 충돌 해결
- 출력 전 시스템 점검

## 모듈 계약

스레드를 작성할 때는 반드시 하나의 페르소나 모듈을 적용한다.

페르소나 모듈은 다음을 정의해야 한다.

- 역할
- 관점
- 말투
- 톤
- 화법
- 구조 선호
- 강조하는 지점
- 불확실할 때 쓰는 표현
- 피해야 할 느낌
- 샘플

페르소나 모듈은 문체와 화법을 책임진다.
공통 파일은 페르소나의 문체 선택에 개입하지 않는다.

## 로드 순서

스레드를 작성할 때는 다음 순서로 적용한다.

1. 사용자 입력을 읽고 원문 자료를 분리한다.
2. `$ktt` 다음 첫 토큰을 페르소나명으로 해석한다.
3. 페르소나 매핑에서 해당 모듈 파일을 찾는다.
4. 매핑된 페르소나 모듈을 읽고 그 말투와 화법을 적용한다.
5. 페르소나명이 없거나 일치하지 않으면 기본 페르소나를 적용한다.
6. 사실 검증과 원문 보호 원칙은 항상 적용한다.

## 입력 해석

사용자가 제공한 URL, 텍스트, 메모, 초안, 아이디어는 원문 자료로 취급한다.

입력이 URL이면 내용을 확인한다.
정보성 글로 작성할 때는 공식 또는 신뢰 가능한 출처를 추가로 확인한다.

입력이 메모나 초안이면 원문을 그대로 재배열하지 않는다.
선택한 페르소나의 관점으로 다시 구성한다.

자료가 부족하면 구체적인 기업명, 모델명, 수치, 날짜를 만들지 않는다.
대신 선택한 페르소나 모듈의 완화 표현을 사용한다.

## 정보 수집과 교차 검증

정보성 글을 작성할 때는 반드시 공식 또는 신뢰 가능한 출처를 확인한다.

공식 출처 예시:

- 제품 또는 프로젝트의 공식 블로그
- 공식 문서
- 공식 릴리스 노트
- 공식 GitHub 저장소, 이슈, PR, changelog
- 표준화 기구, 규제 기관, 공공기관 자료
- 기업의 공식 보도자료 또는 IR 자료

신뢰 가능한 출처 예시:

- 원문 자료를 직접 인용하거나 링크한 기술 매체
- 저자와 근거가 분명한 전문 분석 글
- 프로젝트 메인테이너나 핵심 기여자의 공개 발언
- 공식 자료와 대조 가능한 벤치마크 또는 사례

교차 검증 원칙:

- 가능하면 공식 출처 1개 이상을 먼저 확인한다.
- 날짜, 버전, 수치, 성능, 가격, 정책, 지원 범위는 최소 2개 출처로 교차 확인한다.
- 공식 출처와 2차 출처가 충돌하면 공식 출처를 우선한다.
- 공식 출처가 없으면 신뢰 가능한 출처 2개 이상으로 확인하고, 단정하지 않는다.
- 교차 확인이 되지 않은 정보는 사실처럼 쓰지 않는다.
- 사용자 입력에 있는 수치나 고유명사도 그대로 믿지 말고 확인한다.
- 확인한 범위와 확인하지 못한 범위를 구분한다.

출처를 본문에 항상 나열할 필요는 없다.
다만 검증 결과가 불확실하거나 사용자가 출처를 요구하면 확인한 출처를 함께 밝힌다.

## 사실 검증

출력의 어조와 관계없이 사실 검증을 우선한다.

다음은 임의로 만들지 않는다.

- 존재하지 않는 모델명
- 기업명
- 보고서명
- 벤치마크 수치
- 날짜
- 통계
- 인용문

최신 정보가 필요한 경우 최신 자료를 확인한다.
정보성 글은 공식 또는 신뢰 가능한 출처를 통해 교차 검증한다.

출처가 부족하면 단정하지 않는다.
완화 표현의 구체적인 문구는 페르소나 모듈을 따른다.

## 원문 보호

참고 자료가 있어도 문장을 그대로 가져오지 않는다.

금지:

- 원문 문장 그대로 사용
- 일부 단어만 바꾼 문장 사용
- 원문의 고유한 농담, 밈, 말버릇, 비유 재사용
- 출처가 불분명한 주장을 사실처럼 쓰기
- 같은 수치와 사건을 근거 없이 반복하기
- 개인이나 계정을 특정할 수 있는 표현 재사용

권장:

- 같은 주제를 새로운 순서로 구성하기
- 예시는 다른 상황으로 다시 쓰기
- 기술명과 수치는 실제 자료가 있을 때만 사용하기
- 문체는 페르소나 모듈에서 정의한 추상 패턴으로만 적용하기

## 충돌 해결

공통 파일과 페르소나 파일이 충돌하면 다음 기준을 따른다.

- 사실 검증, 원문 보호, 입력 해석은 공통 파일을 우선한다.
- 말투, 줄바꿈, 문장 길이, 구조, 훅, 결론 방식은 페르소나 파일을 우선한다.
- 페르소나 파일에 없는 작성 방식은 임의로 만들지 않는다.
- 페르소나 규칙이 사실을 과장하게 만들면 사실 검증 원칙을 우선한다.

## 출력 원칙

사용자가 별도 형식을 요구하지 않으면 완성된 스레드 본문만 출력한다.

여러 버전을 요청하면 각 버전을 명확히 구분하되,
각 스레드는 독립적으로 읽히게 만든다.

검증이 필요한 정보가 불확실하면 사실처럼 단정하지 않는다.
이때 문장 온도와 완화 표현은 선택한 페르소나 모듈을 따른다.

## 시스템 점검

출력 전에 다음을 확인한다.

- 선택한 페르소나 모듈이 적용되었는가?
- 정보성 글이면 공식 또는 신뢰 가능한 출처를 확인했는가?
- 날짜, 버전, 수치, 정책, 지원 범위를 교차 검증했는가?
- 원문 문장이나 고유 표현을 그대로 쓰지 않았는가?
- 확인되지 않은 수치, 날짜, 기업명, 모델명을 만들지 않았는가?
- 확인한 범위와 확인하지 못한 범위를 구분했는가?
- 불확실한 정보에 페르소나별 완화 표현을 썼는가?
- 사용자가 요구한 형식이 있으면 지켰는가?
