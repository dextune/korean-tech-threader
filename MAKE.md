# 페르소나 추가 가이드

이 문서는 사용자가 이미지 또는 텍스트로 새로운 말투 자료를 제공했을 때,
그 자료를 분석해 Korean Tech Threader의 새 페르소나로 추가하는 절차를 정의한다.

목표는 특정 사람의 문장을 복제하는 것이 아니다.
자료에서 말투, 화법, 관점, 리듬을 추출해
새로운 한국어 테크 스레드 페르소나 모듈로 재구성하는 것이다.

## 가장 중요한 원칙

제공된 자료의 문장, 문구, 농담, 비유, 고유한 표현을 예시로 그대로 기록하지 않는다.

금지:

- 원문 문장 그대로 사용
- 일부 단어만 바꾼 문장 사용
- 원문의 고유한 농담, 밈, 말버릇, 별명 재사용
- 개인이나 계정을 특정할 수 있는 사례 재사용
- 이미지 속 문장을 OCR한 뒤 예시로 붙여넣기
- 원문과 같은 주제, 같은 순서, 같은 결론으로 샘플 작성

허용:

- 어미, 문장 길이, 단락 리듬 같은 추상 패턴 분석
- 말의 온도, 거리감, 확신 정도 분석
- 관점 전환 방식 분석
- 자주 강조하는 리스크나 가치 분석
- 완전히 다른 주제로 새 샘플 작성
- 플레이스홀더가 들어간 화법 패턴 작성

간단히 말하면, 원문은 분석 재료이고 결과물은 새 문서여야 한다.

## 페르소나 이름 규칙

사용자가 `$ktt <페르소나명>`으로 호출하는 명칭은 반드시 한글로 정한다.

페르소나명은 매우 짧아야 한다.
가능하면 그 페르소나를 함축적으로 의미하는 단어 1개로 정한다.

좋은 페르소나명:

- 한글 1단어
- 기억하기 쉬운 이름
- 말투나 관점을 압축하는 이름
- `$ktt 이름`으로 입력했을 때 어색하지 않은 이름
- 목록 표에서 바로 읽히는 이름

피해야 할 페르소나명:

- 두 단어 이상으로 길어진 이름
- 설명문에 가까운 이름
- 특정 개인, 계정, 회사, 커뮤니티를 직접 떠올리게 하는 이름
- 원문 제공자의 닉네임이나 고유 별명
- 영어, 숫자, 기호가 섞여 호출하기 불편한 이름

페르소나 파일명은 `references/thread-<module-name>.md` 형식을 따른다.
`<module-name>`은 영어 소문자와 하이픈을 우선 사용한다.

파일명은 다음 순서로 정한다.

1. 페르소나 한글명을 자연스러운 영어 단어로 옮길 수 있으면 영어로 쓴다.
2. 영어로 옮기면 의미가 흐려지거나 애매하면 한글 발음을 로마자로 쓴다.
3. 로마자 표기가 길어지면 짧고 읽기 쉬운 형태를 우선한다.
4. 이미 있는 파일명과 겹치면 의미가 살아 있는 짧은 구분자를 붙인다.

예시:

- `거임` -> `references/thread-guim.md`
- `존문가` -> `references/thread-johnmunga.md`
- `세그폴트` -> `references/thread-segfault.md`

한글 호출명과 파일명은 1:1로 매핑되어야 한다.
사용자는 한글 페르소나명만 기억하면 되고,
파일명은 스킬 내부에서 모듈을 찾기 위한 이름이다.

## 기존 페르소나 구조

현재 페르소나 파일은 모두 `references/thread-<module-name>.md`에 둔다.

공통으로 유지할 모듈 구조:

```text
# <페르소나명> 페르소나

이 파일은 Korean Tech Threader의 `<페르소나명>` 말투와 화법을 정의한다.

반드시 `references/thread-common.md`의 공통 모듈 관리 규칙을 먼저 따른다.

이 파일은 다음 작성 요소를 정의한다.

- 말투
- 관점
- 강조하는 리스크 또는 가치
- 독자에게 말을 거는 방식

공통 모듈 계약, 사실 검증, 원문 보호 원칙은 바꾸지 않는다.

## 역할
## 관점
## 말투
## 톤
## 화법
## 구조 선호
## 강조하는 지점
## 불확실할 때 쓰는 표현
## 피해야 할 느낌
## 샘플
```

`## 구조 선호`는 필수 섹션이다.
단락 수, 줄바꿈, 문장 길이, 전개 흐름, 결론 방식을 이 섹션에서 정의한다.

필요하면 페르소나 구현에 필요한 보조 섹션을 추가할 수 있다.
단, 보조 섹션을 늘리는 이유가 말투 구현에 직접 도움이 되어야 한다.

## 분석 절차

새 자료를 받으면 바로 페르소나 파일을 쓰지 말고 먼저 다음을 추출한다.

1. 자료의 표면 특징을 본다.
   - 존댓말인지 반말인지
   - 문장이 짧은지 긴지
   - 줄바꿈이 잦은지 문단이 긴지
   - 질문형, 명령형, 단정형 중 무엇이 많은지

2. 말의 온도를 본다.
   - 차분한지 공격적인지
   - 친근한지 거리감이 있는지
   - 확신이 강한지 유보가 많은지
   - 설명하는 사람인지 발견을 던지는 사람인지

3. 관점의 중심을 찾는다.
   - 무엇을 먼저 보는지
   - 무엇을 과소평가한다고 보는지
   - 어떤 리스크를 반복해서 강조하는지
   - 어떤 가치를 좋다고 여기는지

4. 전개 방식을 찾는다.
   - 첫 문장에서 무엇으로 멈추게 하는지
   - 중간 반전이 어디서 나오는지
   - 사례, 구조, 비용, 감정 중 무엇으로 설득하는지
   - 마지막을 교훈, 감상, 행동, 경고 중 무엇으로 닫는지

5. 재현하면 안 되는 것을 분리한다.
   - 개인 고유의 말버릇
   - 특정 커뮤니티 내부 표현
   - 실명, 회사명, 제품명, 사건명
   - 원문에만 붙어 있는 농담이나 비유

6. 테크 스레드용으로 추상화한다.
   - 원문 주제가 테크가 아니어도 말투 구조만 가져온다.
   - 샘플은 AI, 개발, 인프라, 자동화, 스타트업, 생산성 중 하나로 새로 쓴다.
   - 원문 내용과 같은 소재를 쓰지 않는다.

## 이미지 자료 처리

이미지 자료가 들어오면 먼저 텍스트를 읽거나 OCR한다.
다만 OCR 결과는 내부 분석용으로만 사용한다.

이미지 기반 분석에서 기록할 수 있는 것:

- 줄 길이와 줄바꿈 패턴
- 문장부호 사용 습관
- 짧은 문장과 긴 문장의 비율
- 강조 표현의 위치
- 말투의 거리감
- 시각적으로 반복되는 구성

기록하면 안 되는 것:

- 이미지 속 문장 원문
- 이미지 속 고유한 캡션
- 계정명, 닉네임, 실명, 댓글 내용
- 특정 게시물의 사건 전개를 그대로 옮긴 샘플

## 샘플 작성 규칙

샘플은 페르소나의 말투를 확인하기 위한 것이다.
원문 작성자의 문체를 박제하는 공간이 아니다.

샘플을 쓸 때는 다음 원칙을 지킨다.

- 원문과 다른 테크 주제를 선택한다.
- 실제 기업명, 제품명, 수치가 필요하면 검증된 것만 쓴다.
- 검증이 귀찮으면 일반화된 기술 주제로 쓴다.
- 원문에서 본 비유를 쓰지 않는다.
- 원문과 같은 문장 순서로 전개하지 않는다.
- 너무 닮았다고 느껴지면 더 추상화한다.

안전한 샘플 주제:

- 배포 자동화
- 로그와 관측 가능성
- 로컬 개발 환경
- 모델 서빙 비용
- 문서화와 지식 관리
- 프론트엔드 빌드 구조
- 작은 자동화 실험

## 스킬에 연결하는 절차

새 페르소나 파일을 만든 뒤 다음 파일을 갱신한다.

1. 한글 1단어 페르소나명을 정한다.
2. 영어 단어 또는 한글 발음 로마자로 `<module-name>`을 정한다.
3. `references/thread-<module-name>.md`를 추가한다.
4. `SKILL.md`의 `페르소나 매핑`에 추가한다.
   - 형식: ``- `<한글 페르소나명>` -> `references/thread-<module-name>.md` ``
5. `README.md`의 페르소나 수 배지를 갱신한다.
6. `README.md`의 모듈 구조 트리에 새 페르소나 파일을 추가한다.
7. `README.md`의 모듈 역할 표에 새 페르소나를 추가한다.
8. `README.md`의 말투 예시에 새 샘플을 추가한다.

`SKILL.md`는 런타임 지침만 담기 때문에 목록 표나 긴 예시를 추가하지 않는다.
`README.md`의 Mermaid 흐름도는 페르소나가 늘어나도 `Various Personas` 단일 노드로 유지한다.

## 검증 체크리스트

추가 후 다음을 확인한다.

- `rg "<페르소나명>|thread-<module-name>" -n`
- `git diff --check`
- 페르소나 호출명이 한글 1단어인가?
- 파일명이 영어 단어 또는 한글 발음 로마자인가?
- 한글 호출명과 파일명이 `SKILL.md`에서 1:1로 매핑되는가?
- 새 페르소나 파일이 `references/` 아래에 있는가?
- `SKILL.md`의 `페르소나 매핑`이 실제 파일과 일치하는가?
- `README.md`의 페르소나 수가 실제 개수와 맞는가?
- `README.md`의 모듈 구조 트리와 모듈 역할 표가 실제 파일과 맞는가?
- 샘플에 제공 자료의 문장이나 고유 표현이 들어가지 않았는가?
- 샘플이 Korean Tech Threader의 공통 주제 안에 들어오는가?
- 불확실한 사실을 단정하지 않도록 완화 표현이 있는가?

## 최종 판단 기준

좋은 페르소나 문서는 다음 조건을 만족한다.

- 말투를 따라 쓸 수 있을 만큼 구체적이다.
- 원문 작성자를 떠올릴 만큼 직접적이지 않다.
- 공통 모듈 관리 규칙과 충돌하지 않는다.
- 테크 주제로 변환했을 때 자연스럽다.
- 금지 표현보다 권장 패턴이 더 많다.
- 샘플이 새로 쓴 글처럼 보인다.

애매하면 원문과 더 멀어지는 쪽을 선택한다.
