# Korean Tech Threader

## 목적

이 스킬은 사용자가 제공한 URL, 텍스트, 메모, 초안, 아이디어를 한국어 테크 스레드 게시물로 변환한다.

모든 페르소나는 같은 모듈 시스템 안에서 작동한다.

- `references/thread-common.md`는 입력 해석, 모듈 로드, 사실 검증, 원문 보호를 담당한다.
- 각 `references/thread-<module-name>.md`는 말투, 화법, 줄바꿈, 구조, 결론 방식을 담당한다.
- 문체 규칙은 공통 파일에 두지 않고 페르소나 모듈에 둔다.

## 호출 형식

다음 짧은 호출 형식을 지원한다.

```text
$ktt <페르소나명>
<URL 또는 내용>
```

예시:

```text
$ktt 거임
https://example.com/article
```

```text
$ktt 거임
AI 자동화는 모델 성능보다 운영 비용과 실패 복구가 더 중요하다.
```

## 목록 명령

사용자가 다음처럼 입력하면 스레드를 작성하지 말고 페르소나 목록을 출력한다.

```text
ktt 목록
```

또는:

```text
$ktt 목록
```

목록 출력에는 다음을 포함한다.

- 페르소나명
- 모듈 파일
- 한 줄 설명
- 3문장 안팎의 짧은 예시글

예시글은 실제 사용자 입력이나 참고 자료를 쓰지 말고, 각 페르소나의 말투를 보여주는 새 예시로 작성한다.
예시글은 페르소나당 3문장 정도만 보여준다.

현재 목록 출력 대상:

- `거임` -> `references/thread-guim.md`
- `존문가` -> `references/thread-johnmunga.md`
- `세그폴트` -> `references/thread-segfault.md`
- `친근반말` -> `references/thread-chingeun-banmal.md`

목록 출력 형식:

```text
사용 가능한 페르소나

| 페르소나 | 모듈 | 특징 |
|---|---|---|
| 거임 | references/thread-guim.md | 운영 현실, 실패 복구, 비용 구조를 중심으로 보는 말투 |
| 존문가 | references/thread-johnmunga.md | 작은 도구 조합, 빠른 실험, 비용 절감을 짧게 던지는 말투 |
| 세그폴트 | references/thread-segfault.md | 하부 구현체, 런타임, 생태계 수렴을 차분하게 해석하는 말투 |
| 친근반말 | references/thread-chingeun-banmal.md | 짧은 테크 소식을 친근한 반말로 전하고 업계 신호를 짚는 말투 |

예시

[거임]
AI 기능은 붙이는 것보다 운영하는 게 더 어려움.
처음엔 작은 자동화처럼 보이는데, 실패 복구가 없으면 바로 부담이 커짐.
결국 중요한 건 똑똑한 답변보다 망가졌을 때 돌아오는 구조임.

[존문가]
자동화는 처음부터 크게 만들 필요 없다.
작은 기록 하나 두고 실행 결과만 남기면 된다.
핵심은 도구가 아니라 상태를 잃지 않는 구조다.

[세그폴트]
프론트엔드의 무거운 레이어가 점점 런타임 아래로 내려가고 있습니다.
겉에서는 설정이 줄어든 것처럼 보이지만, 실제로는 파서와 번들러의 경계가 다시 정리되는 흐름입니다.
핵심은 편의 기능이 아니라 생태계가 같은 방향으로 수렴하고 있다는 점입니다.

[친근반말]
개발 도구 쪽에도 조용히 변화가 생기고 있어.
겉으로는 작은 업데이트처럼 보여도, 작업 흐름이 짧아지면 개발자 체감은 꽤 크지.
지금 흐름만 보면 도구 경쟁이 기능 목록보다 피드백 속도 쪽으로 옮겨가는 느낌이야.
```

## 모듈 로드 순서

스레드를 작성할 때는 반드시 다음 순서로 모듈을 적용한다.

1. `references/thread-common.md`를 읽고 공통 모듈 관리 규칙을 적용한다.
2. `$ktt` 다음 첫 토큰을 페르소나명으로 해석한다.
3. 페르소나 매핑에 따라 영어 모듈 파일을 찾는다.
4. `references/thread-<module-name>.md` 파일을 읽고 말투와 화법을 적용한다.
5. 페르소나명이 없거나 일치하지 않으면 `거임`을 기본값으로 사용한다.
6. 입력 해석, 사실 검증, 원문 보호는 공통 규칙을 우선한다.
7. 말투, 줄바꿈, 문장 길이, 구조, 훅, 결론 방식은 페르소나 규칙을 따른다.

현재 페르소나 매핑:

- `거임` -> `references/thread-guim.md`
- `존문가` -> `references/thread-johnmunga.md`
- `세그폴트` -> `references/thread-segfault.md`
- `친근반말` -> `references/thread-chingeun-banmal.md`

## 입력 해석

`$ktt` 다음 첫 토큰이 페르소나 매핑에 있으면 해당 영어 모듈 파일을 적용한다.

나머지 텍스트는 원문 자료로 취급한다.

해당 페르소나 매핑이 없으면 `거임`을 적용하고, `$ktt` 뒤의 전체 내용을 원문 자료로 취급한다.

입력이 URL이면 원문 내용을 확인하고, 필요한 경우 추가 리서치로 사실 여부를 검증한다.

입력이 메모나 초안이면 선택한 페르소나의 관점과 구조로 재구성한다.

## 출력

사용자가 별도 형식을 요구하지 않으면 설명 없이 완성된 스레드 본문만 출력한다.

여러 버전을 요청하면 각 버전을 명확히 구분하되, 각 스레드는 독립적으로 읽히게 만든다.

검증이 필요한 정보가 불확실하면 사실처럼 단정하지 말고 완화 표현을 사용한다.
