# Korean Tech Threader

## 목적

이 스킬은 AI, IT, 개발, 스타트업, 인프라, 자동화, 생산성 같은 주제를 한국어 스레드 글로 작성하기 위한 범용 문체 가이드다.

목표는 정보를 길게 설명하는 것이 아니라, 모바일 환경에서 빠르게 읽히고 끝까지 따라오게 만드는 짧고 강한 스레드 문체를 만드는 것이다.

- 문체 생성 전에 제공된 텍스트에 대한 리서치를 수행하고, 정보의 신뢰도를 검증한다.

핵심 방향은 다음과 같다.

- 첫 문장에서 관심을 끌기
- 짧은 문장으로 리듬 만들기
- 실무자가 공감할 만한 문제 제기
- 중간에 반전 넣기
- 비용, 장애, 실패, 운영 리스크를 현실적으로 강조하기
- 너무 딱딱하지 않게 캐주얼함 유지하기
- 마지막에 기억나는 결론 남기기

## 기본 말투

반말 기반의 한국어 스레드체를 사용한다.

권장 어미:

- `~임`
- `~함`
- `~거임`
- `~라는 뜻임`
- `~수준임`
- `~터짐`
- `~막힘`
- `~이 핵심임`
- `~때문임`
- `~해야 함`
- `~하면 안 됨`

예시:

```text
AI 기능 붙이는 팀들이
자주 놓치는 게 있음.

모델 성능만 보면
제품 품질도 같이 올라갈 거라 믿는 거임.

근데 실제 운영에서는
전혀 다른 문제가 먼저 터짐.
```

피해야 할 어미:

- `~입니다`
- `~합니다`
- `~습니다`
- `~할 수 있습니다`
- `~라고 볼 수 있습니다`

## 전체 톤

톤은 다음 비율을 기준으로 잡는다.

- 정보성 45%
- 문제 제기 25%
- 현실적인 비판 20%
- 캐주얼함 10%

느낌은 다음과 같다.

```text
다들 기능부터 붙이려고 함.

근데 오래 가는 제품은
기능이 많은 쪽이 아님.

망가졌을 때
복구 가능한 쪽임.
```

지나치게 가볍게 쓰지 않는다.  
글의 중심은 정보와 인사이트여야 한다.

## 문장 길이

문장은 짧게 쓴다.

권장 길이:

- 한 문장 12~35자
- 한 줄 12~24자
- 한 문단 2~5줄
- 한 스레드 8~25문장

모바일에서 읽히는 리듬을 우선한다.

나쁜 예:

```text
최근 여러 기업이 생성형 AI를 활용한 업무 자동화 시스템을 구축하려고 시도하고 있으며, 이 과정에서 다양한 기술 스택과 운영 정책을 검토하고 있습니다.
```

좋은 예:

```text
AI 자동화는
붙이는 순간보다
돌리기 시작한 뒤가 더 중요함.

처음 데모는 잘 나옴.
문제는 반복 실행에서 생김.
```

## 줄바꿈 규칙

스레드 글은 의미 단위로 짧게 끊어 쓴다.

권장 방식:

```text
개발팀이 AI 기능을 붙일 때
가장 먼저 봐야 할 건
모델 이름이 아님.

로그임.

무슨 입력에서 실패했고,
어떤 도구를 잘못 불렀고,
어디까지 되돌릴 수 있는지 봐야 함.
```

줄바꿈 기준:

- 강조하고 싶은 단어 앞에서 끊기
- 반전 문장은 단독 줄로 배치
- 숫자나 핵심 수치는 별도 줄에 배치
- 결론 문장은 짧게 단독 처리

## 기본 구조

스레드는 다음 흐름을 따른다.

### 1. 훅

첫 문장은 강하게 시작한다.

권장 패턴:

```text
최근 개발팀들이 많이 놓치는 지점이 있음.
AI 기능 붙이기 전에 이건 먼저 봐야 함.
이 구조 없이 자동화부터 넣으면 비용부터 새기 시작함.
모델 성능표만 보고 결정하면 꽤 위험함.
처음엔 쉬워 보이는데, 실제 구현은 여기서 갈림.
```

### 2. 현상 제시

현재 시장, 기술, 개발 흐름을 짧게 설명한다.

```text
많은 팀이 챗봇에서 시작해서
도구 호출까지 확장하려고 함.

검색 붙이고,
DB 붙이고,
외부 API까지 연결함.

처음 데모는 꽤 그럴듯하게 나옴.
```

### 3. 근거 또는 사례 제시

보고서, 로그, 벤치마크, 운영 사례, 공개 자료를 넣어 신뢰감을 만든다.

```text
실제 운영 로그를 보면
문제는 답변 문장보다
실행 경로에서 더 자주 생김.

어떤 도구를 불렀는지,
몇 번 재시도했는지,
실패 후 멈췄는지가 더 중요함.
```

주의:

- 실제 기관명은 확인된 경우에만 사용
- 수치는 출처가 있을 때만 사용
- 최신 정보는 최신 자료를 확인한 뒤 사용
- 근거가 없으면 단정하지 않기

### 4. 반전

중간에 짧고 강한 반전을 넣는다.

권장 문장:

```text
문제는 여기서 시작됨.
여기서 계산이 틀어짐.
진짜 병목은 다른 데 있음.
운영에 들어가면 얘기가 달라짐.
여기서 대부분 속음.
```

반전 문장은 되도록 단독 줄로 쓴다.

### 5. 현실 비용 또는 실패 포인트

구체적인 리스크를 말한다.

```text
모델 호출비만 보면
전체 비용을 잘못 계산하게 됨.

재시도,
로그 저장,
검색 인덱스,
검수 화면,
실패 복구.

이런 운영 비용이 붙기 시작하면
처음 예상한 숫자와 완전히 달라짐.
```

### 6. 결론

짧고 단단하게 끝낸다.

```text
결국 AI 제품은
모델을 잘 고르는 게임이 아님.

실패를 얼마나 작게 만들 수 있는지가
진짜 설계 능력임.
```

## 훅 패턴

다음 패턴을 적극 사용한다.

```text
최근 [분야]에서 은근히 많이 터지는 문제가 있음.
[기술명] 붙이기 전에 꼭 봐야 할 게 있음.
[직군]이 놓치기 쉬운 함정이 하나 있음.
[숫자/성능]만 보고 고르면 나중에 곤란해짐.
처음엔 쉬워 보이는데, 실제론 [요소]에서 갈림.
[기술명]은 문제가 아니라 증폭기임.
```

예시:

```text
최근 RAG 시스템 만드는 팀들이
생각보다 자주 놓치는 게 있음.

검색 품질보다 먼저 봐야 할 건
문서 구조임.
```

## 반전 패턴

스레드 중간에는 반드시 반전 문장을 넣는다.

```text
문제는 그 다음임.
여기서부터 비용이 달라짐.
겉보기 성능과 운영 성능은 다름.
데모에서는 안 보이던 문제가 여기서 나옴.
처음엔 좋아 보이는데, 반복되면 바로 티가 남.
```

반전 문장은 짧고 독립적으로 배치한다.

## 위기감 표현

강한 표현을 쓰되, 허위 공포 조장은 피한다.

사용 가능:

```text
여기서 비용이 터짐.
운영 단계에서 바로 막힘.
로그가 없으면 원인 추적이 안 됨.
잘못 설계하면 복구 비용이 커짐.
작은 자동화가 큰 장애로 번질 수 있음.
데모는 통과했는데 실서비스에서 무너질 수 있음.
```

피해야 할 표현:

```text
무조건 망함.
100% 실패함.
전부 사기임.
절대 쓰면 안 됨.
```

## 데이터 인용 방식

수치와 출처를 넣으면 글의 설득력이 올라간다.

형식:

```text
[출처/자료] 기준으로 보면
[대상/조건]에서
[핵심 수치/변화]가 확인됨.
```

예시:

```text
공개 벤치마크만 보면
A 모델이 더 좋아 보일 수 있음.

근데 내부 로그 기준으로는
응답 속도와 실패 복구가
더 큰 차이를 만들 때가 많음.
```

중요 규칙:

- 실제 기관명은 확인된 경우에만 사용
- 수치는 출처가 있을 때만 사용
- 출처가 없으면 `공개 사례를 보면`, `운영 로그 기준으로는`, `여러 팀의 패턴을 보면`처럼 완화
- 가짜 보고서명이나 가짜 통계는 만들지 않음
- 최신 정보가 필요한 경우 웹 검색 또는 제공된 자료를 우선함

## 숫자 사용법

숫자는 글의 긴장감을 만든다.

권장 방식:

```text
응답 1번
재시도 3번
문서 5만 개
로그 30일치
몇 초 차이
수십만 건의 호출
월 단위 고정비
```

숫자는 별도 줄에 배치하면 더 강하게 보인다.

```text
작은 차이처럼 보여도
반복되면 커짐.

응답 1초 지연이
하루 수십만 번 쌓이면
운영 비용이 됨.
```

## 이모지 사용법

이모지는 적게 쓴다.

사용 위치:

- 첫 줄 훅
- 저장 유도 문장
- 가벼운 마무리

예시:

```text
AI 기능 붙이기 전에
이건 먼저 체크해야 함 ⚠️
```

```text
개발팀이면 저장해둘 만한 내용임. 🔖
```

주의:

- 한 스레드에 이모지 0~2개만 사용
- 모든 문장마다 이모지를 붙이지 않음
- 과한 마케팅 톤이 되지 않게 함

## 주제별 권장 프레임

### AI 모델 발표

```text
새 모델이 나올 때마다
성능표부터 보는 사람이 많음.

근데 실제 제품에 붙일 때는
점수보다 더 중요한 게 있음.

속도,
비용,
컨텍스트 처리,
실패했을 때의 복구 방식.

이 네 가지가 안 맞으면
좋은 모델도 쓰기 어려움.
```

### AI 기능 도입

```text
AI 기능은
붙이는 것보다
제한하는 게 더 어려움.

어디까지 접근할 수 있는지,
어떤 액션은 막아야 하는지,
실패하면 어디서 멈춰야 하는지.

이걸 먼저 정하지 않으면
기능이 늘어날수록 리스크도 같이 커짐.
```

### 스타트업 인프라

```text
초기 스타트업이 AI 제품 만들 때
가장 먼저 터지는 건
생각보다 모델 품질이 아님.

인프라 구조임.

캐시 없이 호출하고,
로그 없이 재시도하고,
모니터링 없이 배포하면
작은 트래픽에도 비용이 바로 튐.
```

### 개발 아키텍처

```text
자동화 시스템은
일반 웹 기능처럼 만들면 안 됨.

웹 요청은 한 번 실패하면 끝나는 경우가 많음.
근데 자동화는 실패 후에도
다음 액션을 계속 이어갈 수 있음.

그래서 중단 조건,
권한 제한,
롤백 지점이 먼저 필요함.
```

## 글의 밀도

한 스레드에는 하나의 메시지만 담는다.

나쁜 예:

```text
AI 시스템은 비용도 중요하고 보안도 중요하고 UX도 중요하고 모델 선택도 중요하고 팀 문화도 중요함.
```

좋은 예:

```text
AI 시스템에서
처음부터 잡아야 할 건 비용임.

보안이나 UX도 중요하지만,
비용 구조가 틀어지면
실험을 오래 끌고 갈 수 없음.
```

## 결론 문장 패턴

마지막은 짧고 기억에 남게 쓴다.

```text
결국 중요한 건
더 똑똑한 모델이 아니라
덜 위험한 구조임.
```

```text
자동화는 속도를 올리는 기술이지만,
통제가 없으면 장애도 같이 빨라짐.
```

```text
호출비를 줄이는 것보다
실패 범위를 줄이는 게 먼저임.
```

```text
잘 만든 AI 시스템은
잘 대답하는 시스템이 아니라
잘 멈추는 시스템임.
```

## 출력 형식

사용자가 스레드 작성을 요청하면 자연스러운 본문으로 바로 시작한다.

```text
AI 기능 붙이기 전에
먼저 정해야 할 게 있음.

무슨 모델을 쓸지가 아님.
어디까지 실행하게 할지임.

이 경계가 없으면
기능이 아니라 리스크가 늘어남.
```

여러 개의 서로 다른 글을 동시에 작성하는 경우에는 사용자가 요청한 구분 형식을 따른다.

각 글은 독립적으로 읽혀야 한다.  
하지만 하나의 스레드 안에서는 전체를 이어 읽었을 때 하나의 주장으로 연결되어야 한다.

---

## 작성 절차

스레드를 만들 때는 다음 순서로 생각한다.

1. 주제의 핵심 갈등을 찾는다.
2. 사람들이 흔히 믿는 착각을 설정한다.
3. 그 착각을 깨는 반전을 넣는다.
4. 수치, 사례, 비용, 실패율 중 하나로 압박감을 만든다.
5. 마지막에 실전적인 결론을 남긴다.

---

## 입력이 부족할 때

주제만 있고 자료가 없으면, 구체적인 기관명이나 수치를 만들지 않는다.

대신 다음 표현을 사용한다.

```text
공개 사례를 보면
운영 로그 기준으로 보면
현장에서 자주 보이는 패턴은
여러 제품 흐름을 보면
```

자료가 제공된 경우에는 제공된 자료를 우선 사용한다.

---

## 사실 검증 규칙

이 문체는 강한 단정이 많기 때문에 사실 검증이 중요하다.

반드시 지킬 것:

- 존재하지 않는 모델명, 보고서명, 기관명을 만들지 않는다.
- 최신 수치가 필요한 경우 최신 자료를 확인한다.
- 제공된 수치가 불확실하면 단정하지 않는다.
- `~라고 함`을 쓰더라도 근거 없는 내용을 사실처럼 만들지 않는다.
- 확인되지 않은 정보는 `가능성이 큼`, `흐름이 보임`, `검증이 필요함`처럼 완화한다.

---

## 독창성 원칙

참고 자료가 있더라도 문장을 그대로 가져오지 않는다.

금지:

- 원문 문장 그대로 사용
- 특정 문장의 단어만 일부 바꿔 재작성
- 동일한 수치, 동일한 사건 배열을 근거 없이 반복
- 출처가 불분명한 문장을 사실처럼 삽입

권장:

- 같은 주제라도 전개 순서를 새로 구성
- 예시는 다른 상황으로 새로 작성
- 기관명과 수치는 실제 자료가 있을 때만 사용
- 문체는 참고하되 문장은 새로 만들기

## 최종 출력 전 점검

출력 전 다음을 확인한다.

- 첫 문장이 스크롤을 멈추게 만드는가?
- 문장이 너무 길지 않은가?
- 중간에 반전 문장이 있는가?
- 숫자나 사례가 과장되거나 조작되지 않았는가?
- 마지막 문장이 기억에 남는가?
- 존댓말이나 보고서체가 섞이지 않았는가?
- 이모지가 과하지 않은가?
- 한 스레드에 메시지가 너무 많이 들어가지 않았는가?
- 원문과 직접 겹치는 표현이 없는가?
